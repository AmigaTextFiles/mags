2. Häufig gestellte Fragen zum AmigaOS 3.5

»Amiga OS 3.5 - Häufig gestellte Fragen

Dieser Bereich gibt Antworten und Lösungen zu häufig gestellten Fragen. Der
Bereich wird jedes Mal aktualisiert wenn das Supportteam Informationen zu
einem neuen allgemeinen Problem erhält. Schauen Sie daher immer zuerst hier
vorbei, wenn Sie ein Problem haben und lesen Sie die einzelnen Bereiche
aufmerksam durch.


* Vorbereitung zur Installation

Machen Sie unbedingt eine Sicherheitskopie Ihrer
Systempartition.

Möglicherweise müssen Sie ihre Systempartition auch vergrößern um OS 3.5
installieren zu können. Es werden mindestens 20MB freier Festplattenplatz
abgeprüft.

Während des Starts von AmigaOS wird die startup-sequence und auch die
user-startup ausgeführt. Sollten Sie dort Programme starten, die tiefer
ins AmigaOS eingreifen, so deaktivieren Sie diese zuerst. Die Installation
von AmigaOS 3.5 nimmt keine Änderungen an diesen Dateien vor.

68040 bzw. 68060.library

Bei der kompletten Neuinstallation von OS3.5 auf eine leere
Festplattenpartition ist es notwendig, daß OS3.1 zuvor installiert wird.
OS3.1 finden Sie ebenfalls auf der CD-ROM. Beachten Sie dabei jedoch, daß
eventuell für den Betrieb notwendige 68040- oder/und 68060-Bibliotheken
von den jeweiligen Original-Disketten installiert werden müssen, damit Ihr
System von dieser neu installierten Festplattenpartiton wieder gestartet
werden kann.

Am einfachsten erzeugen Sie sich eine Notfalldiskette, booten über diese
Diskette und installieren dann Ihr System neu. Dabei werden sowohl die
notwendigen CPU-Bibliotheken als auch ein eventuell vorhandener
Grafikkartentreiber automatisch installiert.


* Installation

F: Ich erhalte die Fehlermeldung "Sie benötigen mindestens 10
MByte freien Speicher auf Ihrer Bootpartition." habe jedoch 15 MB frei,
doch die Installation bricht ab!

A: Dies ist ein Fehler im Installationsscript. Benötigt werden 20 MB freier
Speicher auf der Bootpartition. Bitte richten Sie diese entsprechend ein
oder entfernen Sie Dateien.

SetPatch

Das Programm SetPatch wird in der startup-sequence gestartet und korrigiert
in der Haupsache bekannte Fehler im AmigaOS. Ab OS 3.5 ersetzt es zudem die
Treiber für die Ansteuerung der internen SCSI und IDE Hardware der
verschiedenen Modelle. Diese neuen Treiber erlauben den Betrieb von
Festplatten mit einer Kapazität größer als 4 Giga Byte.

Es sind Probleme bekannt mit einigen Amiga 3000 Modellen und einigen
4fach-Erweiterungskarten für IDE. In diesem Fall erweitern Sie den Aufruf
des SetPatch Programms bitte folgendermaßen:

C:SetPatch QUIET SKIPROMUPDATES "scsi.device"

Bedenken Sie, dass auch OS 3.5 keine direkte Unterstützung für
4fach-Adapter für IDE enthält. Bitte benutzen Sie die gleiche Software,
die Sie auch bislang verwendet haben.

Sollten Sie dennoch Probleme mit dem Start von AmigaOS 3.5 haben, so
benennen Sie die Datei "DEVS:AmigaOS ROM Update" um. Dadurch werden die
Ersatzmodule insgesamt ausgelassen.

ACHTUNG: WIRD SETPATCH BEIM STARTEN DEAKTIVIERT, STEHT WEDER DIE NEUE ICON-
NOCH DIE WORKBENCH.LIBRARY ZUR VERFÜGUNG.

SetPatch mit NewStyleDevice Patch

Das Programm SetPatch erweitert viele bekannte Hardwaretreiber um neue
Möglichkeiteiten. Dazu benutzt es eine Konfigurationsdatei
"DEVS:NSDPatch.cfg". Uns ist ein Konflikt zwischen dem Hardwaretreiber
cd.device, der Teil des
AmigaOS ist und einem weiteren cd.device (Hersteller unbekannt) gemeldet
worden. Sollten Sie dieses cd.device benutzen, so deaktivieren Sie bitte
die entsprechende Zeile in DEVS:NSDPatch.cfg, in dem Sie mit einem
Texteditor vor die Zeile ein # Zeichen stellen.

LoadV43Module

Wenn Sie Ihr System schon durch den Befehl LoadV43Module und passende
Moduldateien für den Betrieb von Festplatten gößer 4 Gigabyte erweitert
haben, so sollten Sie diese Befehle entfernen. SetPatch übernimmt jetzt
diese Aufgaben.

NSDPatch

Der Aufruf von NSDPatch ist nicht mehr nötig, da SetPatch diese Aufgabe
übernimmt.

NewIcons

AmigaOS 3.5 enthält eine neue icon.library. Diese Bibliothek ist für die
Verwaltung und Darstellung der Piktogramme verantwortlich. Viele Benutzer
haben "NewIcons" installiert, einen Patch, der eine Verbesserung der
Piktogrammdarstellung bewirkt. Die Wirkung dieses Patches wird nun auch
durch die icon.library übernommen. Sie können und sollten den Aufruf des
NewIcon Patches aus Ihrer startup-sequence oder user-startup entfernen.

Das Programm IconEdit kann benutzt werden, um NewIcons in echte OS 3.5
Icons zu verwandeln. Das ist nicht notwendig, um NewIcons auf der
Workbench zu sehen, beschelunigt allerdings das Laden und Darstellen der
Piktogramme.

Füllstandsanzeiger

Die Workbench zeigt Ihnen nun - wie schon zu Zeiten von OS 1.3 - den
Füllstand Ihrer Festplatte links im Fenster an. Diese Anzeige wird jedoch
auch von einigen Patches erwirkt. Deaktivieren Sie diese Patches, sonst
erhalten Sie die Füllstandanzeige doppelt.

ARQ

ARQ (ein Patch, der die Systemrequester verschöndert) hat Probleme auf
manchen OS 3.1 Systemen bereitet. Für OS 3.5 scheinen mehr Systeme
betroffen zu sein. Bitte ersetzten Sie diesen Patch durch einen anderen.

MagicWB

Wollen Sie weiterhin MagicWB Piktogramme benutzen, so benötigen Sie
weiterhin einen Patch, der die Farbpens dafür reserviert. Sonst werden die
Piktorgramme mit falschen Farben dargestellt.

FastIPrefs

Wenn Sie das Programm FastIPrefs benutzen, so sollten Sie dieses wieder
durch IPrefs ersetzen. IPrefs von AmigaOS 3.5 unterstützt neue
Voreinsteller, die OS 3.1 nicht kannte und unterstützt ebenfalls einige
Optionen von FastIPrefs. Ebenso wurden bekannte Fehler entfernt.

Keine Glow Icons nach der Installation

Wenn nach der Hauptinstallation keine Glow Icons sichtbar sind liegt das an
einem Fehler im verwendeten Filesystem. Das Phänomen wurde unter
SmartFileSystem bereits korrigiert. Bitte verwenden Sie die aktuellste
Version. Aber unter PFS3 kommt es vor, daß unser Tool, das die Piktogramme
ersetzt (MergeIcons) keine Dateien zum Ersetzen findet.

Hierbei hilft nur das manuelle Einsetzen der Piktogramme mit den neuen
Workbench-Eigenschaften.

Rahmen um die Ikons kann nicht abgeschaltet werden

Siehe Punkt "FastIPrefs"


* RTG
  Picasso 96
  CyberGraphX CyberGraphX und Picasso96

OS 3.5 unterstützt die Graphikartensoftware CyberGraphX und Picasso96. Für
den Betrieb von CyberGraphX ist es jedoch nötig, daß Sie das Update der
Version 3, welches auf der OS 3.5 CD ROM enthalten ist installieren. Sonst
arbeitet OS 3.5 nicht in jedem Fall fehlerfrei.

Nutzer der Version 4.0 von CyberGraphX laden sich bitte zum Update auf V4.1
die Datei biz/patch/cgxv42_r1.lha aus dem Aminet oder von der "Freie 
Updates"-Seite der Schatztruhe.


* Diverses

Binary Datatype

Durch die Installation von AmigaOS 3.5 werden sämtliche Datatypes
deaktiviert. Nach der Installation können Sie diese wieder aktivieren.
Leider ist der Binary.datatype nicht mit OS 3.5 kompatibel. Dieser
Datatype nimmt insofern eine Sonderrolle ein, daß er keine
Datatype-Beschreibung in DEVS:Datatypes benötigt. Allein die Existenz des
Datatypes in SYS:Classes/Datatypes/ reicht aus, damit er benutzt wird.
Sollten Sie Abstürze bei der Anzeige unbekannter Dateien (Dateien ohne
geeigneten Datatype) in Multiview haben, so stellen Sie bitte sicher, dass
Sie keine Datei Binary.datatype in der Schublade SYS:Classes/Datatypes
haben.

SKIPROMMODULES (Fehler im Handbuch)

Unter der Fehlerbeschreibung im Installationshandbuch wir leider für das
Abschalten des neuen SCSI-Devices eine falsche Kommandozeile-Option
angegeben. Die Richtige Kommandozeile muß heißen:

SetPatch QUIET SKIPROMUPDATES scsi.device

Das einfache Umbennenen der Datei "DEVS:AmigaOS3.5 ROM Update" führt zum
gleichen Resultat.

Absturz beim Öffnen einer Schublade auf der Workbench

Offensichtlich hängt dieser Fehler mit der installierten Picasso 2-Software
zusammen. Bitte installieren Sie CyberGaphX oder Picasso96 welches Sie auf
der CD-ROM finden.

Prefs Programme können nicht gestartet werden

Offensichtlich ist auf Ihrem System ein altes ClassAct in LIBS:
installiert. Bitte kontrollieren Sie, ob Boopsi-Classen in der Schublade
LIBS:Classes enthalten sind und deaktivieren Sie sie. Führen Sie danach
einen Neustart durch und probieren es erneut. Eventuell existiert auch
eine Schublade Sys:ClassAct, die ebenfalls deaktiviert werden muß.

Im Datei-Infofenster erscheint der Einsteller für "Start durch:" nicht

In Ihrer WBStartup-Schublade ist das Tool RAWBInfo enthalten. Durch die
Installation dieses Tools wird der Info-Dialog der Workbench durch einen
neuen Dialog ersetzt, der die ReAction-Eigenschaften nutz. Leider sind
dann allerdings die neuen Workbench-Einstellmöglichkeiten nicht mehr
vorhanden. Bitte deaktivieren Sie dieses Programm durch erneutes Starten
und entfernen es aus der WBStartup-Schublade.

Die AWeb-Toolbar enthält nur Pixelmüll

Dieser Effekt ist auf AGA-Screens zu beobachtem. Leider gibt es zu diesem
Problem noch keine Lösung. Auf OCS/ESC und Grafikkarten-Screens tritt
dieses Problem nicht auf.

Wordworth 7

Wordworth7 startet nicht mehr, da die Art des aufrufenden Piktogramms nicht
korrekt interpretiert wird.
Abhilfe: Wordworth-Piktogramm anklicken, "Informationen anzeigen" aufrufen
und den "Start durch:" von ARexx auf Workbench ändern. Danach läßt sich
Wordworth wieder einwandfrei starten.«
