1. Little helpers for everyday Amiga usage (by Christian Aichinger)

It is hard to believe, but the year 2000 is already almost over. Every day it gets darker, colder and wetter! Wasn't it the same one year ago? Yes, I think I remember it was. But now it is real fun again to sit here in the cozy warmth and write a few reports from the Amiga utility front. So welcome to the little helpers in the December issue of Amiga Aktuell.

First of all thank you for all the nice feedback I got on the last issue. If you are trying to flatter me, I like that!! Keep it coming :). For this issue I picked the programs BlazeWCP, Next 3.0, PowerWindowsNG, HappyEnv and the GUI creator "r" from Aminet. Feeling comfortable? Candle lit? Cup of tea ready? Let's get going.

Next 3.0:

Next 3.0 is mainly a display program for all kinds of text and hypertext documents. Compared to the ridiculously small file size of not even 40 KB, the feature list is immense (so there are still assembler programmers ;)).

Next 3.0 can display, print and save the following text file formats: ASCII text in DOS, Macintosh and Amiga/UNIX formats, AmigaGuide, HTML, word, write, WordPerfect, RTF, FTXT (text from Clipboard), HLP, man (UNIX help files), Finalwriter/Wordworth documents and finally even exotic files like MacBIN and MacDOC. I hope I have forgotten none!? Files unknown to Next 3.0 can be displayed too, of course, but control characters will be ignored or the text will be displayed unformatted (if it is a text at all).

Text documents can be saved as ASCII text or printed. When printing, you can use several format algorithms to achieve a good printout.

Files which have been compressed with XPK, LHA, LZX, ZIP, RAR or Powerpacker, or for which a Datatype exists, can be displayed by Next 3.0, too, but for this to work, you will need the appropriate system extensions or decrunchers. To display a file using Datatypes, you first have to activate the DT-Mode. Then you can view all files Multiview can display (including graphics, sounds and animations).

Hypertext documents like AmigaGuide or HTML are displayed so that you can follow local hyperlinks with a mouse click as usual. But it doesn't much support gimmickry like inlined graphics, text flow or HTML frames.

But the speed Next shows while layouting and displaying such files is really amazing. The HTML document "book-main.html" from the AmigaOS 3.5 CD ROM (in the directory "Manual/dos", the file size is 1.079.617 bytes) is displayed in less than five seconds (using a 68040/40MHz reading from a 32x-SCSI-CD-ROM drive). The common Amiga web browsers demand more patience from the user.

The handling is somewhat unusual. The scroll bar which is used to move through the document is located at the lower end of the window and is used horizontally - not the usual vertically on the left or right side of the window. The cursor keys as well as the number pad are used the usual way (cursor up/down = up/down, 9/3 = page up/down, 7/1 = top/bottom, etc.)

Typical Amiga control elements like menus or gadgets are found only very seldom or not at all.

Right clicking opens a window where you can select available options from a text list. But this text list is a little unordered.

To make up for that, Next 3.0 requires very little RAM. The program itself only needs about 80 KB of free RAM, and if the document is too big for the free RAM of the computer, it is just loaded as it is displayed.

To be able to use the many functions Next 3.0 offers, you should remember the most important hotkeys, and read the extensive documentation, which is available in German and English. Next 3.0 is a program for users who think that function, flexibility and speed are more important than super smooth scrolling and graphical gimmickry.

BlazeWCP:

BlazeWCP is a small patch program that replaces the slow OS functions for C2P conversion (C2P="chunky to planar") WriteChunkyPixelLine8() and WritePixelArray8() by faster versions.

The new routines are on average twice as fast, and under optimal conditions even more than five times as fast as the original functions of the OS or functions of similar programs. If BlazeWCP is used under AmigaOS 3.1, it also replaces the buggy function WriteChunkyPixels() by a bug free routine. Older patchers like "NewWPA8" or "PatchWPA8" are now superfluous and can or should be removed from the system.

Besides the patch itself, the distribution also contains English documentation and the program "wpatest", which can be used to measure the speed increase. Before starting "wpatest", you should have read the included README file, because there are some problems, especially with screens that have a resolution lower than 640x480 pixels.

To use BlazeWCP, you need at least an 68020 processor, free 32-bit Fast RAM and OS 3.0. The AGA chipset is not necessary, but useful. BlazeWCP makes use of the patch collection "FBlit" by Stephen Brookes if it is present. An installer file is not included - to install the patch, copy the file BlazeWCP into the directory "C": and add an appropriate entry into the file "startup-sequence". If you have the software "patchcontrol" installed, BlazeWCP should be started after "patchcontrol". Programs that profit from using BlazeWCP are some AVI- and QuickTime-players, some web browsers, datatypes for graphics, emulators and some other programs. You can find more details in the manual.

r:

No, this is not a typo. That's really the name of the program ;). If you know the program "Argue", you will have an idea what "r" is trying to do. "r" creates graphical user interfaces for any shell program. Unlike Argue, it doesn't use MUI, but the "ordinary" gadtools.library of the AmigaOS, which in turn means that it doesn't use up a lot of system RAM. There's not much to install. You just have to unpack the program directory somewhere on your hard disk. Then just set a path to the directory, or even better, copy "r" to the "C:" directory.

Attention! The distribution contains two different versions in the directory "bin". One is for OS 2.1 and one for OS 3.0. So you should make sure that you install the correct version. The catalogs directory contains some localizations (deutsch, english, françcais, svenska, norsk). Unfortunately, "r" doesn't work at all if there is no catalog installed. So pay attention to the short well-written German or English AmigaGuide manual, especially the section "installation".

So far, so good. Now that "r" is installed, we could take a look at the GUI of the command "list". In a shell, enter: 

r list

On OS 3.5 systems, this window should open:

Nice, isn't it? This works with almost all shell programs that follow the Amiga-Style-guide. To be more specific: Those that use the "ReadArgs()" templates as they were defined by Commodore-Amiga for OS 2.0. If the user has written a batch file or an ARexx script which uses the "ReadArgs()" templates, too, "r" works just as well. If you start the batch file "S:DPAT" with: 

r "execute s:dpat"

you get this GUI:

But "r" loses it with programs that define their arguments in a non-standard way. This mostly means old programs from Kick1.3 and quick-and-dirty ports from UNIX/BSD/M$ platforms (like ports of 3D shooters like Quark, Duum and Hermetic), but sadly also some new programs like "mpega".

PowerWindows(NG):

One more patch. Some people might already know about PowerWindows, since some other publications have already written about it. But time doesn't stand still, and a lot of things have changed in this program.

PowerWindows is an OS patch that enables the user to move windows beyond the borders of the screen (as known from MacOS and Windows), to iconify windows, to move them with their contents ("Opaque-Move"), and some other things.

In the -for patch- very extensive distribution you'll find two different binaries. One is called PowerWindows, the other PowerWindowsNG. The NG-version requires the ROM remapper Blitzkick or a patched ROM image of the AmigaOS. A ROM module for Blitzkick is included. This module changes some ROM routines in a way that makes it "safer" to use PowerWindowsNG. Also included (Thank god!) is a preferences editor for changing the appearance and functionality of PowerWindows.

This can also be done via shell arguments. But configuring using shell would be real work, since the preferences editor already requires a lot from the user. A help function that explains each button is integrated, but unfortunately this explanation is sometimes less clear than the label on the button itself. So in any case you should read the English manual. And even this doesn't always help, because Georg Steger's knowledge of English is at least as good as mine ;). But most functions are understandable, and a few others can be interpreted by using some imagination even without the manual.

But it's going to take you some time to configure PowerWindowsNG so that your system is as stable as it was without the patch. All the functions of PowerWindows are less than useful if the computer keeps freezing in "stress situations". You have to check all your programs for bad behavior: this includes fast resizing of windows, low memory situations, of course "moving the window out of the screen" and the behavior of screens with different color depths (e.g. AGA and 8bit screens). If a program has problems, it can be "left out" in the PowerWindows preferences.

The documentation says PowerWindows needs an Amiga with OS 2.0 (probably) and is compatible with the common graphic card drivers. A fast CPU, 32-bit Fast RAM and other hardware to speed up the computer won't harm either. PowerWindows, at least the tested version, is compatible with OS 3.5.

Sadly, Georg Steger has stopped development of this software because he wanted to invest more time in the AROS project. But he has released it as OpenSource to the Amiga community. The manual sometimes still says that PowerWindows is Shareware, but the README to the archive has this corrected.

HappyENV:

And finally a real classic. HappyENV is a little older, which makes me wonder even more how I never read anything about it. Time to do something about this.

HappyENV replaces the assign "ENV:", which is normally only a directory in the "Ram Disk", by a "real" drive. This drive is comparable with the "Ram Disk", but the "HappyENV" drive has some advantages. In the "Ram Disk", environment variables always require at least 1KB (1024 Bytes), even though most variables are much smaller. Some are as small as 1 Byte. If those small files take up a thousandfold of their size in RAM, it is an enormous waste of space. Furthermore, all environment variables are copied from "ENVARC:" to "ENV:" during bootup, even if the program that uses a variable is not running, or has been uninstalled. When using "HappyENV", variables are not copied until programs ask for them. This noticeably speeds up the booting process, especially if, over time, several megabytes have accumulated in the "ENVARC:" assign.

If a program (be it the Workbench, the shell, or during the bootup) needs a certain environment variable, HappyENV intercepts the access to "ENV:". Then it looks inside "ENVARC:" to find the file or directory and copies it to the drive created by HappyENV. Then the calling program works on as if nothing happened. The user doesn't notice it at all, but he has a lot of free RAM for more important things and the computer boots faster. Isn't this something?

Installation is relatively easy. The directory of HappyENV contains, besides the English documentation and the source directory, two files required for installation.

The file "HappyENV-Handler" has to be copied to the directory "L:", and "MountENV" is best stored in the directory "C:".

Now for the (somewhat) more complicated part. In the startup-sequence, you have to find the section where the directory "Ram:ENV" is created and the environment variables are copied from "ENVARC:" to "ENV:". These entries have to be removed first. In the original startup-sequence of OS 3.5 these entries look like this: 

C:MakeDir >NIL: RAM:T RAM:Clipboards RAM:ENV RAM:ENV/Sys
C:Copy >NIL: ENVARC: RAM:ENV ALL NOREQ

The second line ("C:Copy >NIL:...") has to be commented out or removed completely, and first line must be changed to this: 

C:MakeDir >NIL: RAM:T RAM:Clipboards

Then look below for the line... 

Assign >NIL: ENV: ENVARC:

... and replace it by... 

C:MountENV

... and you're done. If you could not make any sense of the instructions above, try to find someone to help you. The HappyENV-documentation explains it more thoroughly, too.
